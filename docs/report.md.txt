# Calendar QA System - Project Report

## Project Overview
A Question-Answering system built with Rust and the Burn framework that reads calendar documents and answers questions about dates, events, and committee meetings.

**GitHub Repository**: https://github.com/MeiZ-7715/word-doc-qa

---

## Section 1: Introduction

### Problem Statement
The goal of this project was to build a Question-Answering system that can process Word documents containing academic calendar information and answer natural language questions about their content. The system needed to handle questions like "When is the graduation ceremony in 2026?" and "How many times did the HDC hold their meetings in 2024?"

### Motivation
Educational institutions generate extensive documentation about schedules, events, and meetings. Having an automated Q&A system helps students, faculty, and staff quickly find information without manually searching through multiple documents.

### Approach Overview
I implemented a transformer-based neural network using the Rust Burn framework. The system:
1. Loads and parses .docx calendar files
2. Extracts structured information about dates, events, and committees
3. Uses a transformer model with embedding layers
4. Provides a CLI interface for asking questions

### Key Design Decisions
- **Burn Framework**: Rust-native deep learning for performance and safety
- **Transformer Architecture**: Embedding layers for efficient text processing
- **Rule-Based Fallbacks**: Pattern matching for reliable answers to common questions
- **BPE Tokenizer**: Byte-Pair Encoding for consistent text tokenization

### Architecture Details

#### Layer Specifications
| Component | Input Size | Output Size | Parameters |
|-----------|-----------|------------|------------|
| Token Embedding | 30,522 | 128 | 3,906,816 |
| Positional Embedding | 512 | 128 | 65,536 |
| Output Projection | 128 | 30,522 | 3,906,816 |
| **Total** | | | **7,879,168** |

#### Key Components
1. **Token Embedding**: Converts input token IDs into dense 128-dimensional vectors
2. **Positional Embedding**: Adds sequence position information (up to 512 tokens)
3. **Output Projection**: Maps final representations to vocabulary size for answer generation

### Data Pipeline

#### Document Processing
1. **Read .docx files** using `docx-rs` library
2. **Extract text** from paragraphs and runs
3. **Parse entities** using regex patterns:
   - Dates: `JANUARY 2024`, `15`, `2026`
   - Committees: `Higher Degrees Committee`, `Senate`
   - Events: `GRADUATION`, `HOLIDAY`, `MEETING`
4. **Generate QADocument** structure with extracted data

#### Tokenization Strategy
- **Tokenizer**: BPE (Byte-Pair Encoding) from `tokenizers` crate
- **Vocabulary size**: 30,522 tokens
- **Special tokens**: `[CLS]`, `[SEP]` for question-context separation
- **Maximum length**: 512 tokens
